Tageszeit-Verwaltung

Hilfe > Tageszeit-Verwaltung

Übersicht

Die Tageszeit-Verwaltung ist ein hierarchisches System zur Definition und Verwaltung von Arbeitszeiten in HCC Plan. Es ermöglicht die flexible Konfiguration von Tageszeiten auf verschiedenen Ebenen: Projekt, Person, Location und Planungsperioden.

Hauptfunktionen

  • Hierarchische Struktur - Projekt → Person/Location → Planungsperioden
  • Kategorien-System - Tageszeit-Kategorien (Morgen, Nachmittag, Abend) definieren
  • Standard-System - Bestimmte Zeiten als Standards für Vererbung markieren
  • Flexible Zeitdefinition - Start- und Endzeiten präzise festlegen
  • Reset-Funktion - Zurücksetzen auf übergeordnete Standards
  • Multi-Ebenen-Verwaltung - Verschiedene Dialoge für verschiedene Kontexte

System-Architektur

🏗️ Hierarchische Vererbung

Projekt: Globale Tageszeit-Definitionen
Person/Location: Spezifische Anpassungen
Planungsperioden: Periodenbezogene Feinabstimmung

Die drei Dialog-Systeme

1. Tageszeit-Kategorien verwalten

Dialog: DlgTimeOfDayEnum

Zweck: Definition der grundlegenden Kategorien wie "Morgen", "Nachmittag", "Abend"

  • Name - Vollständige Bezeichnung der Kategorie
  • Abkürzung - Kurze Darstellung für UI
  • Tagesreihenfolge - Numerische Sortierung im Tagesverlauf
  • Neu-Modus - Erstellen neuer Kategorien

2. Einzelne Tageszeit bearbeiten

Dialog: DlgTimeOfDayEdit

Zweck: Definition konkreter Arbeitszeiten innerhalb einer Kategorie

  • Name - Spezifische Bezeichnung (z.B. "Vormittagsprobe")
  • Startzeit - Präzise Anfangszeit
  • Endzeit - Präzise Endzeit
  • Kategorie-Zuordnung - Auswahl der Tageszeit-Kategorie
  • Standard-Option - Als Standard für Vererbung markieren
  • Neu-Modus - Erstellen vs. Bearbeiten

3. Tageszeiten-Verwaltung (Multi-Kontext)

Dialog: DlgTimeOfDaysEditList + Builder-Pattern

Zweck: Verwaltung aller Tageszeiten für verschiedene Entitäten

  • Kontext-adaptive UI - Unterschiedliche Funktionalität je nach Entität
  • Tageszeiten-Tabelle - Name, Zeitspanne, Kategorie, Standard-Status
  • New/Edit/Delete - Vollständige CRUD-Operationen
  • Reset-Funktion - Zurück zu übergeordneten Standards (wo verfügbar)
  • Automatische Sortierung - Nach Tageszeit-Index sortiert

Grundbedienung

Tageszeit-Kategorien einrichten (Projekt-Ebene)

  1. Öffnen Sie die Tageszeit-Kategorien-Verwaltung
  2. Definieren Sie Grundkategorien: "Morgen", "Nachmittag", "Abend"
  3. Vergeben Sie Abkürzungen (z.B. "VM", "NM", "AB")
  4. Setzen Sie Tagesreihenfolge (1, 2, 3...)
  5. Speichern Sie die Kategorien

Projekt-Tageszeiten definieren

  1. Öffnen Sie die Tageszeiten-Verwaltung für das Projekt
  2. Klicken Sie New... für neue Tageszeiten
  3. Geben Sie einen aussagekräftigen Namen ein
  4. Setzen Sie Start- und Endzeit
  5. Wählen Sie die passende Kategorie
  6. Markieren Sie wichtige Zeiten als Standard
  7. Speichern Sie mit OK

Person-spezifische Anpassungen

  1. Öffnen Sie Tageszeiten für eine spezifische Person
  2. Das System zeigt zunächst Projekt-Standards
  3. Passen Sie Zeiten für diese Person an (Edit/Delete)
  4. Fügen Sie person-spezifische Zeiten hinzu (New)
  5. Nutzen Sie Reset um zu Projekt-Standards zurückzukehren

Verwandte Funktionen

  • Stammdaten - Für Person- und Location-Verwaltung mit Tageszeit-Zuweisungen
  • Actor Plan Period - Für personenbezogene Planungsperioden
  • Location Plan Period - Für locationbezogene Planungsperioden
  • Hauptplanung - Für die Verwendung der Tageszeiten in der Terminplanung

Anwendungsbereich

Verschiedene Verwendungsebenen

  • Projekt-Ebene: Globale Tageszeit-Standards für das gesamte Projekt
  • Person-Ebene: Individuelle Arbeitszeiten und Verfügbarkeiten
  • Location-Ebene: Öffnungszeiten und verfügbare Zeiten von Arbeitsstätten
  • Planungsperioden: Feinabstimmung für spezifische Zeiträume

Typische Anwendungsszenarien

  • Theater-Projekt: "Vormittagsprobe 09:00-12:00", "Nachmittagsprobe 14:00-17:00", "Abendvorstellung 19:30-22:00"
  • Workshop-Reihe: "Session 1 09:00-10:30", "Pause 10:30-11:00", "Session 2 11:00-12:30"
  • Festival: "Aufbau 08:00-10:00", "Programm 10:00-18:00", "Abbau 18:00-20:00"

Praktische Anwendungsfälle

🎯 Szenario 1: Neues Projekt Setup

Zu Projektbeginn werden zunächst Kategorien definiert ("Probe", "Vorstellung", "Pause"), dann konkrete Zeiten ("Vormittagsprobe 09:00-12:00") zugeordnet und relevante als Standards markiert.

🎯 Szenario 2: Person mit besonderen Arbeitszeiten

Ein Teilzeit-Mitarbeiter kann nur vormittags. Seine Person-Tageszeiten werden angepasst: Vormittags-Zeiten bleiben, Nachmittags/Abends werden entfernt.

🎯 Szenario 3: Location mit eingeschränkten Öffnungszeiten

Eine Probebühne ist nur bis 20:00 verfügbar. Die Location-Tageszeiten werden entsprechend angepasst, späte Abendzeiten entfernt.

🎯 Szenario 4: Planungsperiode mit Sonderregelung

Während der Intensivproben-Woche gelten andere Zeiten. Die Planungsperioden-Tageszeiten werden temporär angepasst, nach der Woche über Reset zurückgesetzt.

Häufige Probleme & Lösungen

Problem: Tageszeit-Name bereits vorhanden

Ursache: Name-Duplikate sind pro Kontext nicht erlaubt

Lösung: Verwenden Sie eindeutige, beschreibende Namen wie "Vormittagsprobe" statt nur "Probe"

Problem: Tageszeit kann nicht ausgewählt werden

Ursache: Keine Tageszeit-Kategorien definiert

Lösung: Erstellen Sie zuerst Kategorien über die Kategorien-Verwaltung, dann Tageszeiten

Problem: Reset-Button nicht verfügbar

Ursache: Entität ist auf oberster Ebene (Projekt) oder hat keine übergeordneten Standards

Lösung: Reset funktioniert nur bei Vererbungs-Hierarchien (Person/Location erben von Projekt)

Best Practices

💡 Kategorien zuerst definieren

Beginnen Sie immer mit der Definition sinnvoller Tageszeit-Kategorien auf Projekt-Ebene. Diese strukturieren später alle konkreten Zeiten.

🚀 Standards strategisch setzen

Markieren Sie häufig verwendete Tageszeiten als Standards. Diese werden automatisch an untergeordnete Entitäten vererbt und reduzieren den Konfigurationsaufwand.

⚡ Vererbung nutzen

Definieren Sie realistische Standards auf Projekt-Ebene und passen Sie nur bei Bedarf auf Person/Location-Ebene an. Dies spart Zeit und sorgt für Konsistenz.

🔄 Hierarchie verstehen

Projekt-Standards → Person/Location-Anpassungen → Planungsperioden-Feintuning. Jede Ebene kann die übergeordnete überschreiben.

Erweiterte Funktionen

Builder-Pattern für verschiedene Kontexte

Das System verwendet ein flexibles Builder-Pattern, das für jeden Kontext (Projekt, Person, Location, Planungsperioden) automatisch die richtigen Commands und Vererbungsregeln konfiguriert.

Intelligente Vererbung

  • Projekt → Person: Person erbt Projekt-Standards, kann aber anpassen
  • Projekt → Location: Location erbt Projekt-Standards
  • Person → ActorPlanPeriod: Planungsperiode erbt Person-Einstellungen
  • Location → LocationPlanPeriod: Planungsperiode erbt Location-Einstellungen

Standard-System

Tageszeiten können als "Standards" markiert werden, wodurch sie automatisch an untergeordnete Entitäten vererbt werden. Dies reduziert den Konfigurationsaufwand erheblich.

Automatische Bereinigung

Das System führt automatisch eine Bereinigung nicht verwendeter Tageszeiten durch (delete_unused) beim Speichern oder Abbrechen.

Technische Details

Builder-Pattern Implementation

Das System verwendet ein abstraktes Builder-Pattern (DlgTimeOfDayEditListBuilderABC) mit spezialisierten Implementierungen für:

  • Project: Globale Projekt-Tageszeiten
  • Person: Person-spezifische Arbeitszeiten
  • Location: Location-spezifische Verfügbarkeiten
  • ActorPlanPeriod: Planungsperioden-spezifische Anpassungen für Personen
  • LocationPlanPeriod: Planungsperioden-spezifische Anpassungen für Locations

Command-Pattern Integration

Jeder Builder konfiguriert automatisch die korrekten Commands für CRUD-Operationen in seinem spezifischen Kontext, inklusive Undo/Redo-Funktionalität.
